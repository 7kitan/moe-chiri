Thinking in Components

2026.06.14

designoat-uicode

A component is not a file. It is a boundary. The boundary between what changes together and what changes separately. Most teams get this wrong. They split by file type — components/, hooks/, utils/ — instead of by responsibility.

The Wrong Abstraction

I have seen a Button.tsx with fifteen props. Variant, size, loading, iconLeft, iconRight, fullWidth, onClick, href, and seven more. That is not a component. That is a config file wearing a trench coat.

A good component does one thing and hides the rest. It exposes a small surface area. You should be able to look at its props and understand what it does without reading the implementation.

Composition over Configuration

Oat UI takes this approach. Instead of a mega-Select with twenty props, you compose:

<Select.Root>
  <Select.Trigger>
    <Select.Value />
  </Select.Trigger>
  <Select.Content>
    <Select.Item value="1">Option</Select.Item>
  </Select.Content>
</Select.Root>

This pattern — borrowed from Radix — gives you control without overwhelm. Each sub-component owns its own styling and behaviour. You reach for what you need and leave the rest.

A component is a boundary. The boundary between what changes together and what changes separately.

What You Own

The other principle is ownership. A component should own its state, its layout, and its side effects. If a concern leaks out — if the parent needs to know about inner DOM structure or internal state — the boundary is broken.

I wrote about this in the Oat UI docs as the "three gates" rule: a component owns its data, its rendering, and its behaviour. Anything that crosses those gates should be explicit in the props.

Components are not about reuse. They are about isolation. Reuse is a side effect.
